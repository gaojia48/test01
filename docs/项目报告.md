# 基于 DeepSeek 的 Linux 运维 Agent Skills 系统项目报告

## 1. 项目背景

Linux 系统管理中经常需要执行磁盘检查、日志分析、进程排查、网络诊断和日常巡检。传统方式通常依赖管理员手动输入多条命令，例如 `df`、`du`、`grep`、`ps`、`ss`、`journalctl` 等。对初学者来说，命令多、参数多、输出复杂，容易出现不会查、查错方向或误执行危险命令的问题。

本项目围绕 Linux 系统管理与 Shell 编程课程要求，设计并实现一个 Linux 运维 Agent Skills 系统。用户可以用自然语言描述问题，系统通过 DeepSeek 判断是否需要触发运维 skill；如果触发，则执行预先登记的 Shell 脚本；如果没有触发，则只进行普通回答，不执行任何系统命令。

项目强调：

- 能在 Linux 环境真实运行。
- 能解释每个 Shell 脚本和 Linux 命令的作用。
- 能通过测试验证核心功能。
- 能通过 Git 记录体现开发过程。
- AI 只辅助理解和总结，不直接执行任意命令。

## 2. 需求分析

### 2.1 功能需求

系统需要支持以下功能：

- 自然语言输入运维问题。
- 自动选择合适的 Agent Skill。
- 执行受控 Shell 脚本。
- 生成 Markdown 诊断报告。
- 查看报告列表和最新报告。
- 清理项目生成的报告。
- 环境自检，检查 Python、依赖、API Key、代理、Linux 命令和脚本权限。
- DeepSeek 不可用时提供本地关键词匹配模式。

### 2.2 非功能需求

- 安全性：禁止 AI 直接生成并执行任意 Shell 命令。
- 可维护性：Python 逻辑、YAML skill、Shell 脚本分层管理。
- 可测试性：核心模块有自动化测试。
- 可演示性：提供交互入口 `./ops` 和演示流程文档。
- 可迁移性：部署到其他 Linux 时可通过 `doctor` 快速检查环境。

## 3. 系统设计

### 3.1 总体架构

系统采用分层架构：

```text
用户输入
  -> main.py / ops 交互入口
  -> Planner 判断是否触发 skill
  -> Skill Loader 读取 YAML 配置
  -> Executor 执行受控 Shell 脚本
  -> Report 生成 Markdown 报告
  -> DeepSeek 负责意图识别和结果总结
```

项目目录结构：

```text
linux-ops-agent/
  main.py
  ops
  config.yaml
  agent/
  skills/
  scripts/
  reports/
  docs/
  tests/
```

### 3.2 Agent Skills 设计

每个 skill 是一个明确的运维能力，使用 YAML 文件描述。例如 `disk_check` 表示磁盘检查能力，配置中包含：

- skill 名称
- 功能描述
- 对应脚本
- 风险等级
- 关键词
- 允许使用的命令

这种设计的好处是：DeepSeek 只能从已有 skills 中选择，不能随意构造系统命令。

### 3.3 安全执行设计

系统执行命令时遵循以下规则：

- 只执行 `scripts/` 目录下预设脚本。
- 默认只包含低风险诊断命令。
- 对删除、格式化、关机、重启等危险请求直接拒绝。
- 没有匹配到 skill 时不默认执行巡检。
- 每次脚本执行有超时限制。

## 4. 功能实现

### 4.1 交互入口

项目提供 `./ops` 作为便捷入口。用户可以直接进入交互模式：

```bash
./ops
```

在交互模式中可以输入：

```text
检查磁盘空间问题
分析最近 SSH 登录失败
reports
last report
clear reports
doctor
exit
```

这样避免每次输入 `python main.py`，也避免中文参数必须加引号的问题。

### 4.2 DeepSeek 接入

系统通过 OpenAI 兼容方式接入 DeepSeek：

```text
DEEPSEEK_API_KEY
DEEPSEEK_MODEL
DEEPSEEK_BASE_URL
```

DeepSeek 的作用主要有两个：

- 根据用户自然语言选择 skill。
- 根据 Shell 脚本输出生成诊断报告。

如果 API Key 缺失、依赖缺失、网络异常或代理配置异常，系统会输出诊断提示，并回退到关键词模式。

### 4.3 Shell 脚本功能

项目目前包含 5 个运维脚本：

- `disk_check.sh`：检查磁盘空间、inode、日志目录占用和大文件。
- `log_analyze.sh`：分析认证日志、SSH 失败登录和系统错误。
- `process_check.sh`：检查负载、内存、CPU 和高占用进程。
- `network_check.sh`：检查 IP、路由、监听端口和网络连通性。
- `health_report.sh`：综合调用多个检查脚本，生成巡检输出。

这些脚本体现了课程要求中的 Shell、Linux 命令、管道、文本处理、日志分析和自动化流程。

### 4.4 报告管理

每次诊断会在 `reports/` 目录生成 Markdown 报告。交互模式中支持：

- `reports`：查看报告列表。
- `last report`：查看最新报告内容。
- `clear reports`：清理报告文件。

### 4.5 环境自检

`doctor` 命令用于检查部署环境：

```bash
./ops doctor
```

检查内容包括：

- Python 版本
- Python 依赖
- DeepSeek API Key 是否读取
- 代理环境变量
- Linux 命令是否存在
- Shell 脚本是否有执行权限
- skills 配置是否能加载

## 5. 测试与验证

项目使用 pytest 进行自动化测试：

```bash
pytest
```

测试覆盖：

- skill 配置加载。
- planner 根据关键词选择 skill。
- DeepSeek 无 skill 时可返回普通回答。
- 危险请求拒绝。
- executor 安全执行脚本。
- report 报告生成。
- latest report 辅助函数。

人工验证命令：

```bash
./ops doctor
./ops 检查磁盘空间问题
./ops 分析最近 SSH 登录失败
./ops 0
./ops --no-llm
```

## 6. AI 工具使用说明

本项目允许使用 AI，但不把 AI 作为不可控执行者。AI 的边界如下：

- 可以理解用户意图。
- 可以从预设 skills 中选择工具。
- 可以总结命令输出。
- 不可以直接生成任意 Shell 命令并执行。
- 不可以执行删除、格式化、关机、重启等危险操作。

这种方式能兼顾 AI 的自然语言理解能力和 Linux 运维的安全性。

## 7. Git 开发过程

项目使用 Git 管理代码。开发过程采用多次小提交，每次提交对应一个明确功能，例如：

- 初始实现 Linux 运维 Agent 系统。
- 设置运维脚本可执行权限。
- 改进 DeepSeek 环境诊断提示。
- 修复 DeepSeek 代理兼容问题。
- 添加 ops 便捷入口和交互模式。
- 未匹配请求不再默认执行 skill。
- 添加报告管理命令。
- 添加最新报告查看命令。
- 添加运行环境自检命令。

实际提交记录可通过以下命令查看：

```bash
git log --oneline
git log --oneline --stat
```

## 8. 总结与改进

本项目完成了一个可运行、可解释、可测试、可维护的 Linux 运维 Agent Skills 系统。项目结合 DeepSeek、Python、Shell 脚本和 Linux 命令，实现了从自然语言输入到自动化诊断报告生成的完整流程。

后续可继续改进：

- 增加更多 skill，例如服务检查、防火墙检查、安全配置检查。
- 增加更多测试日志样本。
- 增加 Web 页面作为辅助展示。
- 支持多机器 SSH 巡检。
- 为高风险操作加入人工确认流程。

当前版本已经能够满足课程项目对 Linux 系统管理、Shell 编程、自动化、Agent 工具调用、测试和文档的要求。
